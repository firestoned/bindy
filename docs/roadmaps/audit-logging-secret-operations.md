## Overview

This plan implements structured audit logging for all Kubernetes Secret operations in the bindy operator to meet compliance requirements for the regulated banking environment. Currently, the operator performs Secret operations (creation, rotation, deletion, updates) without dedicated audit trails, creating compliance gaps.

**Why this is needed:**
- **Compliance Requirement**: Banking/regulated environments require auditable logs of all sensitive resource access
- **Security Monitoring**: Track which component accessed Secrets and when
- **Incident Response**: Provide forensic trail for security investigations
- **Change Tracking**: Understand when and why Secrets were rotated, deleted, or modified

**Current State:**
- RNDC Secret operations in `src/reconcilers/bind9instance/resources.rs` (creation, rotation, deletion)
- DNSSEC Secret references in `src/bind9_resources.rs` (volume mounts, env vars)
- General structured logging exists via `tracing` crate
- No dedicated audit logging target or compliance-friendly output

**Intended Outcome:**
- All Secret operations have structured audit logs with operation type, resource metadata, and outcome
- Audit logs filterable via dedicated `target: "audit"`
- Both JSON (machine-readable) and text (human-readable) formats supported
- Zero performance impact when audit logging disabled
- Secret data values never logged — only metadata
- Comprehensive documentation for compliance teams and operators

---

## Design: Hybrid Audit Module + Tracing Integration

Use a dedicated audit module (`src/audit.rs`) providing:
- Type-safe audit event enums (`SecretAuditOperation`, `SecretGenerationType`)
- Helper functions for common operations (`audit_secret_create`, `audit_secret_rotate`, etc.)
- Structured field logging with dedicated `target: "audit"` for filtering
- Integration with existing `tracing` infrastructure

**Why this approach:**
- Minimal code disruption — only audit calls added at operation points
- Compliance-friendly — dedicated target enables separate log sinks (Splunk, ELK, CloudWatch)
- Extensible — easy to add ConfigMap or CRD auditing later
- Consistent with existing logging patterns

## Implementation Phases

### Phase 1: Core Audit Infrastructure

#### Step 1.1 — Create `src/audit.rs`

Create module with:
- `SecretAuditOperation` enum: `Create`, `Delete`, `Update`, `Read`, `Rotate`
- `SecretGenerationType` enum: `AutoGenerated`, `UserSupplied`, `InlineSpec`
- `SecretAuditContext` struct holding all audit metadata
- `audit_secret_operation()` — main logging function using `target: "audit"`
- Helpers: `audit_secret_create()`, `audit_secret_delete()`, `audit_secret_rotate()`, `audit_secret_update()`

**Key requirements:**
- NEVER log Secret data values (keys, passwords, etc.)
- Include: namespace, name, component, operation type, success/failure
- Support additional context via `HashMap` (rotation count, algorithm, etc.)
- Structured field logging for JSON/text compatibility

#### Step 1.2 — Export audit module in `src/lib.rs`

```rust
pub mod audit;
```

#### Step 1.3 — Update `initialize_logging()` in `src/main.rs` (lines 64-96)

- Enable `with_target(true)` to expose log targets for filtering
- Default `EnvFilter`: `"info,audit=info"`
- Add `AUDIT_LOG_ENABLED` environment variable support (default: `true`)
- Emit startup message: `info!(target: "audit", "Audit logging enabled")`

**Environment Variables:**

| Variable | Default | Description |
|----------|---------|-------------|
| `AUDIT_LOG_ENABLED` | `true` | Enable/disable audit logging |
| `RUST_LOG` | `info,audit=info` | Log level filter |
| `RUST_LOG_FORMAT` | `text` | Output format: `json` or `text` |

## Log Output Examples

### JSON Format (Production / SIEM)

```json
{
  "timestamp": "2026-03-09T10:15:30.123456Z",
  "level": "INFO",
  "target": "audit",
  "fields": {
    "message": "Secret operation completed successfully",
    "operation": "rotate",
    "resource_type": "Secret",
    "namespace": "bindy-system",
    "name": "primary-bind9-rndc-key",
    "component": "bind9instance::rotate_rndc_secret",
    "result": "success",
    "audit.rotation_count": "5",
    "audit.algorithm": "hmac-sha256"
  }
}
```

### Text Format (Development)

```
2026-03-09T10:15:30.123456Z  INFO audit: Secret operation completed successfully
  operation="rotate" resource_type="Secret" namespace="bindy-system"
  name="primary-bind9-rndc-key" component="bind9instance::rotate_rndc_secret"
  result="success" audit.rotation_count="5" audit.algorithm="hmac-sha256"
```

### Filtering Recipes

```bash
# All audit events
kubectl logs deploy/bindy-operator -n bindy-system | jq 'select(.target == "audit")'

# Only rotations
kubectl logs deploy/bindy-operator -n bindy-system | jq 'select(.target == "audit" and .fields.operation == "rotate")'

# All failures
kubectl logs deploy/bindy-operator -n bindy-system | jq 'select(.target == "audit" and .fields.result == "failure")'

# CSV export for compliance report
kubectl logs deploy/bindy-operator -n bindy-system | \
  jq -r 'select(.target == "audit") | [.timestamp, .fields.operation, .fields.namespace, .fields.name, .fields.result] | @csv' \
  > audit_report.csv
```

## Security Constraints

**NEVER logged:**
- Secret data values (RNDC keys, passwords)
- Base64-encoded Secret contents
- DNSSEC private keys
- Any credential material

**Always logged (metadata only):**
- Secret names and namespaces
- Operation type and timestamp
- Rotation counts and algorithm names
- Success/failure status and error messages
- Component performing the operation
